# 复盘报告骨架

> 本文件是写复盘报告时的格式参考。只在实际落盘复盘记录时加载。
>
> 设计原则：报告的核心价值是**还原"为什么这么改"的因果链**和**记下未解决项的去向**，
> 不是把已落盘的改动再抄一遍。改动清单已在各落点文件就地标注来源，报告里只列文件+一行摘要。

---

## 骨架

```markdown
# [主题] 复盘报告

> 日期：YYYY-MM-DD
> 复盘范围：[项目名/任务/时间段]
> 复盘类型：工作区级 / 项目级（见 SKILL.md 阶段五落点判定）

## 一、问题总览

共发现 X 个问题。归因分布：提示词类 N 个、知识库类 N 个、流程类 N 个。
（一句话概括最突出的模式，如"主要集中在输出格式约束缺失"）

## 二、教训（按严重度排序，核心章节）

### 教训1：[一句话标题]（严重度：高/中/低）

**过程**：还原"不理想时刻"的对话轨迹——用户说了什么，AI 怎么做错的，纠了几轮才对齐。
（这是各落点文件里查不到的独占信息，必须记。）

**根因**：为什么会错。1-3 条，关联的根因要列出来，不要只写孤立的一条。

**修复**：改了哪个文件、加了什么规则。
（一句话指向落点即可，不重复抄规则全文——规则已在 AGENTS.md/SKILL.md 就地落盘。）

### 教训2：...

## 三、变更清单（指向落点，不抄内容）

### 提示词变更
| 文件 | 变更摘要 | 对应教训 |
|------|---------|---------|
| AGENTS.md | 加"示例≠定义"约束 | 教训1 |

### 知识库变更 / 技能变更 / 规范变更 / 需求回收
（同上格式，有则列，无则省略整节）

## 四、未解决/推迟项

| 问题 | 暂不处理原因 / 去向 |
|------|-------------------|
| 截图系统无法读图 | 环境限制，用数值验证替代 |
| XX功能推迟 | 已纳入需求池 T3 |

## 五、回归验证（2-3 条）

| 原问题 | 验证方式 | 结果 |
|--------|---------|------|
| 条目遗漏 | 从源文档生成汇报 | ✓ 一致，标注 8/8 |
```

---

## 写作要点

1. **教训叙事是核心**：过程还原 + 根因关联 + 修复落点。只看变更清单看不出"为什么"，教训章节补上因果链。已有的复盘里，scaffold 那份的"教训1/2/3"就是这个写法。

2. **变更清单只列摘要**：`AGENTS.md → 加"示例≠定义"约束` 一行够了。规则全文已在落点文件里，报告不重复抄。

3. **未解决项必须记**：明确判定"本次不处理"的问题（环境限制、超出范围、待外部输入）是独占信息，没有别的落点，只能在复盘里留下判定理由，否则下次复盘会重复发现。

4. **回归验证可简**：2-3 条典型问题即可，不需要全量回归。验证过程口头汇报过，报告里只记结论。
