# 项目复盘 Skill

## 触发条件

用户在对话中提到以下任意关键词时触发：

- "复盘"、"回顾项目"、"优化智能体"、"整理经验"
- "改进工作流"、"项目总结"、"哪里可以提升"
- "这个项目有什么可以优化的"、"下次怎么做得更好"

## 前置条件

在执行复盘前，确认以下信息可用：

- 用户能指出要复盘的项目/任务范围（最近一次、某个具体项目、某个时间段）
- 该项目的对话记录可回顾（当前会话内或用户可提供摘要）

若范围不明确，先追问："复盘哪个项目/任务？最近一次还是某个具体的？"

---

## 阶段一：收集 —— 提取不理想时刻

### 步骤

1. 回顾项目对话记录，逐条标记每个"需要用户介入纠正/补充/解释"的时刻
2. 按以下5类归档：

| 问题类型 | 识别信号 | 记录格式 |
|---------|---------|---------|
| **格式偏差** | 输出少字段、层级错、字数超限、缺少必要模块 | `[格式] 具体偏差描述` |
| **概念盲区** | 用户不得不解释术语、背景、约定 | `[概念] 缺失的概念/术语` |
| **规则漏执行** | 用户手动检查后发现问题（如条数对不上、示例值被写死） | `[规则] 未执行的规则内容` |
| **知识过期** | 引用了旧版本逻辑、已废弃字段/接口 | `[知识] 过期内容 → 正确内容` |
| **流程卡顿** | 一个简单判断需要 ≥3 轮对话才对齐 | `[流程] 卡顿环节描述` |
| **工具链** | 标注不规范导致脚本失败、_index 状态失真、脚手架占位符漏写、build 产物被手改 | `[工具链] 具体问题` |

### 输出格式

```markdown
## 问题清单

| # | 类型 | 问题描述 | 发生场景 | 严重度 |
|---|------|---------|---------|--------|
| 1 | 规则 | 汇报文档条目数对不上源文档 | 生成XX汇报时 | 高 |
| 2 | 流程 | 角色权限边界反复确认3轮 | 定义系统角色时 | 中 |
```

> 严重度：高=每次必现且影响交付质量 / 中=偶尔出现或影响效率 / 低=一次性的小问题

---

## 阶段二：归因 —— 三选一判定

### 决策树

对阶段一的每个问题，走以下判定：

```
问题出现时，智能体…

  本应知道"怎么做"但做错了？
  └─ 提示词问题 → 缺行为约束/校验规则/禁止事项

  根本"不知道"这件事的存在？
  └─ 知识库问题 → 缺事实信息/术语定义/背景约定

  知道怎么做，但每次都要用户手动引导流程？
  └─ 技能缺失 → 缺标准化工作流封装
```

### 归因判断辅助表

| 问题表现 | 大概率是 | 为什么 |
|---------|---------|--------|
| 输出格式/风格不对 | 提示词 | 没有约束输出规范 |
| 不理解项目术语 | 知识库 | 术语未定义 |
| 需要你手动数条目 | 提示词 | 没有校验规则 |
| 每次都要重复同一套引导语 | 技能 | 流程未封装 |
| 引用旧版接口 | 知识库 | 接口文档未更新 |
| 多轮才对齐一个简单判断 | 技能 | 判断流程未标准化 |
| 示例值被写死为定义 | 提示词 | 缺少"示例≠定义"约束 |
| 标注不规范/脚本失败/_index 失真 | 提示词 | 工具链规则未约束或漏执行 |

### 输出格式

在阶段一的问题清单上追加两列：

```markdown
| # | … | 归因 | 修复位置 |
|---|-----|------|---------|
| 1 | … | 提示词 | AGENTS.md（见下方规则分层） |
| 2 | … | 知识库 | 03知识库/术语表 |
| 3 | … | 技能缺失 | 新建 Skill |
```

## 阶段三：提炼 —— 技能候选识别与共性规则抽取

### 3a. 技能候选筛选

对每个"流程卡顿"类问题和你手动引导过的完整工作流，过这个 checklist：

```
□ 本项目内执行了 ≥3 次？
□ 包含 ≥3 个步骤（不是单一操作）？
□ 有明确的触发条件（用户说什么时该触发）？
□ 有稳定的输入→输出映射？
□ 换个新项目还能复用（可参数化）？
□ 标准化后能减少用户的介入次数？
```

- **≥4 个勾** → 值得做成独立 skill
- **2-3 个勾** → 合并到已有 skill 或写为提示词规则
- **≤1 个勾** → 写为提示词规则或知识库条目

### 技能候选评估表

```markdown
## 技能候选评估

| 候选名称 | 重复次数 | 步骤数 | 触发明确 | I/O稳定 | 跨项目复用 | 减少介入 | 得分 | 决定 |
|---------|---------|--------|---------|---------|-----------|---------|------|------|
| 详设→汇报生成 | 5 | 4 | ✓ | ✓ | ✓ | ✓ | 6 | 新建skill |
| 合规差距分析 | 2 | 5 | ✓ | ✓ | ✗ | ✓ | 4 | 合并提示词 |
```

### 3b. 共性规则抽取

对每个提示词类和知识库类问题，先判断"是不是 AI 行为硬约束"，再按三层规范体系分流：

| 规则性质 | 判断信号 | 落地点 |
|---------|---------|--------|
| **AI 行为硬约束**（触发时机+怎么做+禁止做什么） | 涉及 AI 自动执行或禁止动作 | 工作区 `<workspace>/AGENTS.md`（ZCode 自动加载的速查卡） |
| **跨项目通用工程流程规则** | 换项目还会出现 | `01-通用规则/项目工作流规范.md`（完整手册）+ 工作区 `AGENTS.md` 同步一条速查 |
| **项目专属约定** | 仅当前项目适用 | 该项目 `项目工作流约定.md`（无特殊约定则不建该文件） |

> **不在复盘写入范围**：用户级 `~/.config/.../AGENTS.md`（个人偏好、写作风格），复盘时不要动它。
> **不要使用死记忆**：规则只写入上述三层文件，不要写到任何 `.memory`/`MEMORY.md` 路径——那些路径不会被自动加载，写进去等于丢失。

```markdown
## 规则分层

| 规则 | 性质 | 落地点 |
|------|------|--------|
| "示例≠定义"约束 | AI 硬约束 | 工作区 AGENTS.md |
| 新版本复制后必须剥离标注 | 跨项目通用流程 | 01-通用规则/项目工作流规范.md + AGENTS.md 速查 |
| 算力项目基线需求冻结 | 项目专属 | 算力/项目工作流约定.md |
```

## 阶段四：执行 —— 五条线落地

### 4a. 提示词线

需修改的文件和修改内容：

| 层级 | 文件 | 改什么 |
|------|------|--------|
| 工作区级 | `<workspace>/AGENTS.md` | AI 硬约束：触发时机+怎么做+禁止做什么（ZCode 自动加载） |
| 项目级 | `<项目>/项目工作流约定.md` | 项目专属约定（无特殊约定则不建该文件） |

> 用户级 `~/.config/.../AGENTS.md`（个人偏好/写作风格）不在复盘写入范围。

修改原则：
- 每条规则写清楚"什么时候触发 + 怎么做 + 禁止做什么"
- 避免笼统表述（"做好校验"→ 写成"逐条对照源文档条目数，输出时标注 N/N 通过"）

### 4b. 知识库线

```markdown
## 知识库变更

| 操作 | 文件/位置 | 内容 |
|------|----------|------|
| 新增 | 03知识库/政策法规/XX/术语表.md | 补充XX术语定义 |
| 更新 | 03知识库/政策法规/XX/ | 更新转换产物 |
| 过期 | 03知识库/政策法规/XX_旧版.md | 标记已过期，替代：XX_新版_转换.md |
```

### 4c. 技能线

对阶段三确认要做的 skill，创建到 `~/Skills/`（用户级，跨项目共享），按项目工作流规范的 Skill 模板创建。

### 4d. 案例线

每个项目复盘后，确认是否值得沉淀案例：

```
□ 本次踩坑在同类项目中是否会重现？
□ 关键决策是否有通用参考价值？
□ 是否发现了新的工作模式？

≥1 个勾 → 写入 03知识库/工作案例库/对应领域/
        → 更新 工作案例库/_index.md
```

案例至少包含：背景、关键决策、踩坑记录、复用信号。

**案例入库后**，检查该案例是否涉及某个设计基线的使用/踩坑/改造：

```
□ 案例涉及的端态匹配到了哪个设计基线？
  ├── B端管理后台 → 01-通用规则/设计库/web后台-design.md
  ├── 对外门户 → 01-通用规则/设计库/web门户-design.md
  ├── H5/小程序 → 01-通用规则/设计库/手机端-design.md
  └── 移动端极简 → 01-通用规则/设计库/手机端-极简黑白-design.md

→ 在对应基线的"关联案例"表格中追加一行
→ 格式：| 案例名 | 领域编号-领域名 | 关键复用点 |
```

### 4e. 规范层

复盘发现的流程设计缺陷，更新 `01-通用规则/项目工作流规范.md`（遵循 V2.0 四层迭代结构：项目 / 通用 / 迭代 / 版本内目录，非旧的七目录平铺）：

```
□ 某个目录规则在实际使用中不合理？
□ 缺少某个操作场景的标准流程？
□ 文件关联规则有遗漏？
□ 项目阶段是否变更（如 设计→开发）？

≥1 个勾 → 更新规范文档；项目阶段变更 → 同步更新 00-项目总览.md 的"当前阶段"
        → 变更记录标注 来源：[项目名] 复盘 YYYY-MM-DD
```

## 阶段五：验证 —— 回归测试

### 步骤

1. 从阶段一挑 2-3 个最典型的"不理想时刻"
2. 构造同样的输入，用更新后的系统重新执行
3. 检查规则是否兜住、知识是否补全、技能是否减少介入

### 验证记录

```markdown
## 回归验证

| 原问题 | 测试输入 | 期望结果 | 实际结果 | 通过 |
|--------|---------|---------|---------|------|
| 条目遗漏 | 从源文档生成汇报 | 条目数一致 | 一致，标注8/8 | ✓ |
| 角色边界模糊 | 描述四端五角色 | 每个角色带边界 | 已补充 | ✓ |
```

**存放位置**：

## 复盘纪律

复盘本身也必须遵守以下规则：

1. **问题不能丢失**：阶段一的每个问题，在阶段四必须有对应的修复动作（或明确列入"未解决"）
2. **归因不能模糊**：不允许"可能是提示词也可能是知识库"，必须做出判断
3. **技能不滥建**：通过 checklist 过滤，不把"只用过一次"的流程做成 skill
4. **变更可追溯**：每条规则/知识/技能的修改必须记录"为什么改"和"来源：[项目名]复盘 YYYY-MM-DD"
5. **案例不重复**：写入案例库前先查已有案例，同一个踩坑只更新不新建
